# 专业Markdown编写规范指南

## 📋 概述

本指南基于您的Ubuntu项目实际文件结构，提供一套完整的Markdown编写规范，帮助您创建专业、一致、易维护的技术文档。

## 🎯 核心原则

### 1. 一致性原则
- 文档格式保持统一
- 命名规范一致
- 结构模式一致
- 代码示例风格一致

### 2. 可读性原则
- 逻辑清晰，层次分明
- 语言简洁明了
- 代码示例完整可执行
- 术语准确统一

### 3. 可维护性原则
- 模块化组织内容
- 易于更新和扩展
- 版本信息完整
- 引用关系清晰

## 📁 文件组织规范

### 目录结构设计原则

```
项目根目录/
├── docs/                          # 📚 文档主目录
│   ├── guides/                    # 📖 指南和规范
│   ├── basics/                    # 🎯 基础配置 (001-099)
│   ├── automation/                # 🤖 自动化 (100-199)
│   ├── advanced/                  # 🔧 高级配置 (200-299)
│   └── troubleshooting/           # 🔧 故障排除 (900-999)
├── scripts/                       # 🔧 脚本目录
├── examples/                      # 💡 示例和演示
├── history/                       # 📜 历史记录
└── templates/                     # 📄 模板文件
```

### 文件命名规范

#### ✅ 推荐的命名方式

| 类型 | 格式 | 示例 | 说明 |
|------|------|------|------|
| 主文档 | `三位数字-简短描述.md` | `001-basic-setup.md` | 便于排序和识别 |
| 指南文档 | `功能描述-guide.md` | `docker-installation-guide.md` | 明确文档性质 |
| 测试报告 | `功能-test-report.md` | `docker-test-report.md` | 区分测试内容 |
| 模板文件 | `文件名.template.md` | `config-file.template.md` | 标识模板性质 |
| 历史文档 | `功能-说明-日期.md` | `ssh-troubleshooting-20251105.md` | 时间戳标识 |

#### ❌ 避免的命名方式

- `ubuntu_setup_guide.md` (下划线，无序号)
- `SSH问题排查.md` (中文文件名)
- `temp.md` (无意义的名称)
- `README_README.md` (重复名称)

## 📝 Markdown语法规范

### 1. 标题规范

#### ✅ 正确的标题层级
```markdown
# 一级标题 (文档标题)
## 二级标题 (主要章节)
### 三级标题 (子章节)
#### 四级标题 (详细说明)
##### 五级标题 (小节)
###### 六级标题 (最小标题)
```

#### 使用原则
- 一级标题：每篇文档只有一个
- 二级标题：主要功能模块
- 三级标题：具体操作步骤
- 避免跳级使用（如从##直接到####）

### 2. 列表规范

#### 无序列表
```markdown
- 项目1
  - 子项目1.1
  - 子项目1.2
- 项目2
```

#### 有序列表
```markdown
1. 第一步操作
2. 第二步操作
   1. 子步骤2.1
   2. 子步骤2.2
3. 第三步操作
```

#### 任务清单
```markdown
- [x] 已完成任务
- [ ] 待完成任务
- [ ] 进行中任务
```

### 3. 代码块规范

#### 行内代码
```markdown
使用 `command --option` 执行操作
变量名为 `${VARIABLE_NAME}`
```

#### 代码块
```markdown
```bash
# Bash脚本
#!/bin/bash
echo "Hello World"
```

```python
# Python代码
def hello():
    print("Hello World")
```
```

#### 代码块标识符
- `bash` - Shell脚本
- `python` - Python代码
- `java` - Java代码
- `go` - Go代码
- `json` - JSON配置
- `yaml` - YAML配置
- `sql` - SQL查询
- `markdown` - Markdown示例

### 4. 表格规范

#### 标准表格
```markdown
| 功能 | 状态 | 描述 | 优先级 |
|------|------|------|--------|
| Docker | ✅ 完成 | 容器化部署 | 高 |

```

#### 对齐表格
```markdown
| 左对齐 | 居中对齐 | 右对齐 |
|:-------|:-------:|-------:|
| 内容   | 内容   | 内容   |
| 较长   | 较短   | 数字   |
```

### 5. 链接规范

#### 内部链接
```markdown

```

#### 外部链接
```markdown

```

#### 图片链接
```markdown

```

### 6. 强调和格式

#### 文本格式
```markdown
**粗体文本** - 重要内容
*斜体文本* - 强调内容
***粗斜体*** - 特殊强调
~~删除线~~ - 废弃内容
`代码文本` - 命令或变量
```

#### 引用块
```markdown
> 普通引用文本
>
> > 嵌套引用文本
```

#### 分隔线
```markdown

主要内容分隔

---
次要内容分隔
```

## 🎨 内容编写规范

### 1. 语言风格

#### 技术文档写作原则
- **准确性**: 确保技术信息准确无误
- **简洁性**: 用最少文字表达清晰意思
- **完整性**: 提供必要的前置条件和后续步骤
- **一致性**: 术语和格式保持统一

#### 中文技术文档规范
```markdown
# ✅ 推荐表达
- 使用"安装"而非"安裝"（简体中文）
- 使用"配置"而非"设定"
- 使用"用户"而非"使用者"
- 技术术语保持英文原词，如"API"、"CLI"

# ❌ 避免表达
- 中英文混用不规范："这个API很powerful"
- 口语化表达："咱们来搞个docker"
- 过于口语化："搞定了，没问题"
```

### 2. 代码示例规范

#### 完整性要求
```markdown
# ✅ 好的代码示例
```bash
# 1. 更新系统包
sudo apt update

# 2. 安装Docker
sudo apt install -y docker.io

# 3. 启动服务
sudo systemctl start docker

# 4. 验证安装
docker --version
```

# ❌ 不完整的示例
```bash
apt update
apt install docker
docker start
```
```

#### 注释规范
```bash
# 配置Docker镜像加速器
# 解决国内网络访问问题
sudo tee /etc/docker/daemon.json > /dev/null << 'EOF'
{
  "registry-mirrors": ["https://mirror.ccs.tencentyun.com"]
}
EOF
```

### 3. 交互元素规范

#### 复杂操作的分步说明
```markdown
### 配置SSH密钥认证

#### 步骤1: 生成SSH密钥
```bash
# 在本地机器上生成密钥对
ssh-keygen -t rsa -b 4096 -C "your_email@example.com"
```

#### 步骤2: 复制公钥到服务器
```bash
# 方法1: 使用ssh-copy命令
ssh-copy-id user@server

# 方法2: 手动复制
cat ~/.ssh/id_rsa.pub | ssh user@server "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"
```

#### 步骤3: 测试连接
```bash
# 测试SSH连接
ssh user@server
```
```
